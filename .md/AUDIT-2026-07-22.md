# Skeptical audit — `robinhood agentic trading` (2026-07-22)

**Verdict:** The pipeline runs, on schedule, and its numbers are honestly produced by the code as it stands — nothing is fabricated. But the system that runs is not the system the README describes. The README sells an LLM trading agent with hard guardrails on Robinhood; what actually executes every weekday is a z-score mean-reversion backtest ticked forward one day at a time on **free Yahoo data**, with **8 trading days** of forward history. The guardrail/executor/broker architecture — "the core" per the README — is on no scheduled path anywhere. The LLM half runs daily but has produced literally nothing: every one of its 130 recorded decisions is flat, and 113 of the 130 are API failures logged as decisions. Nothing in production has ever touched Robinhood. So: the appearance of a working paper-trading research loop is real; the appearance of a working *agent* is not.

## 1. What this is supposed to do

From the code: (a) `forward.py` appends one day of net return per weekday for the config-locked strategy (`mean_reversion` + conviction gate, 65 mega-caps) to a growing forward record; (b) `papertrade.py` replays that strategy bar-by-bar over the cached window and writes a trade ledger; (c) `runner.py` is a separate, manually-invoked loop where an LLM (or a strategy) proposes orders through code-enforced caps against a mock or MCP broker.

Where the docs disagree with the code:
- `README.md:19` — "How it works: `cron → python -m rhagent.runner`". No cron invokes runner: `crontab -l` returns "no crontab for adijain", and neither workflow in `.github/workflows/` mentions it. The real cron entry point is `scripts/paper_cron.sh`, which never touches runner.
- `README.md:57` and `.env.example:4` — "fill in ANTHROPIC_API_KEY". Nothing in `src/` reads that variable (grep: only `agent.py:129` mentions it, in a stale docstring calling the loop "the real Claude loop"). The code uses `NVIDIA_API_KEY` (`runner.py:122`).
- `README.md` layout table — "`data.py` | Historical bars via the RH MCP". In production the bars come from Yahoo (see §5).
- `guardrails.py:2` — "This module is the heart of the system's safety." It defends a path with zero scheduled traffic.

## 2. The real entry point

Two GitHub Actions workflows, both verified green and recent (`gh run list`: last success 2026-07-22T08:03Z):

```
daily-paper-run.yml  (cron 17 11 * * 1-5, UTC)
  └─ scripts/paper_cron.sh
       ├─ git restore data/ journal/ from origin/paper-state
       ├─ python -m rhagent.refresh --fetch --days 10     → refresh.py → _fetch_yahoo (no MCP secrets in CI)
       ├─ python -m rhagent.forward                       → forward.py → strategies/mean_reversion.py
       │     → overlay.apply_conviction → backtest.net_returns → journal/forward/mean_reversion/returns.csv
       ├─ python -m rhagent.forward --engine agent        → engine.AgentEngine (NVIDIA LLM) + memory.py + learn.py
       └─ git worktree push data/ + journal/ → paper-state branch
  └─ scripts/make_dashboard.py (+ evaluate.py, evaluate_robust.py) → Vercel deploy

research-run.yml  (workflow_dispatch; also triggered by the unauthenticated
  Vercel endpoint deploy_api/trigger-run.js)
  └─ python -m rhagent.papertrade --engine mean_reversion --symbols all
       → papertrade.py → engine.StrategyEngine → overlay.ConvictionGate → journal/papertrade/<run_id>/
```

Files actually on the live path: `paper_cron.sh`, `refresh.py`, `data.py`, `forward.py`, `backtest.py`, `config.py`, `strategies/{__init__,base,mean_reversion}.py`, `overlay.py`, `engine.py`, `learn.py`, `memory.py`, `evaluate.py`, `evaluate_robust.py`, `features.py`, `papertrade.py`, `scripts/make_dashboard.py`, `deploy_api/trigger-run.js`. That's it.

## 3. What is dead

- **The entire README centerpiece**: `runner.py`, `agent.py`, `executor.py`, `guardrails.py`, `broker.py`, `journal.py`, `mcp_session.py`, `strategy_runner.py`. Import graph confirms these eight import only each other; nothing on the live path imports any of them, no cron/workflow invokes runner. They run only when a human types the command (last manual run: today, per `journal/runs.jsonl` — 39 lines total, 14 run_starts ever, 3 of which died before `run_end`).
- **The HALT-file kill switch and daily-loss halt** (`runner.py:24,86`) — only checked by runner, so dead in practice.
- `compare.py` — imported by nothing (grep across `src/` and `scripts/`); manual CLI only.
- `decision_pnl.py` — imported by nothing, including the dashboard. Manual CLI. Its entire dataset is 7 decisions, 5 scorable.
- `strategies/momentum.py`, `strategies/linreg.py` — in the registry but config locks `mean_reversion`; reachable only via manual `--engine` flags.
- `ANTHROPIC_API_KEY` in `.env.example` — read by nothing.
- `journal/forward/pairs/` — empty local leftover, absent from paper-state.
- `learning.md`, `docs/superpowers/plans/` (8 files), `graphify-out/`, `.codex/`, `docs/audit-prompt.md` — notes/artifacts, nothing reads them.
- **Tests are real but no CI runs them.** The only two workflows are the paper runs. All 215 tests pass (verified by running them), but a regression would ship to the daily run unnoticed until it crashed in production.
- **Local repo hazard, not code**: 203 files under `data/` and `journal/` are force-staged on `main` right now (git status "A") despite `.gitignore` saying "never commit account data" and the whole design keeping them on `paper-state`. If someone commits, main diverges from the architecture.

## 4. Does it run?

Yes. Verified by execution, not reading:
- `pytest`: **215 passed in 9.14s**, zero failures.
- `MOCK_AGENT=true python -m rhagent.runner`: dry-run completes; guardrails accept the $250 order and reject the $10,000 one with the correct reason.
- `python -m rhagent.forward --report`: works (local record is a stale 3-day copy; the live 8-day record is on `paper-state`).
- `python -m rhagent.decision_pnl`: works (7 decisions, hit_1d 100% on n=5 — meaningless sample).
- CI: last `daily-paper-run` succeeded 2026-07-22, log shows refresh → tick appended 1 day → push.

The unscheduled LLM runner path needs `NVIDIA_API_KEY` (present in `.env`, `LIVE=false`) and was not exercised live in this audit.

## 5. Where the data really comes from

- **Production: Yahoo, not Robinhood.** The CI log for run 29902520119 shows `ROBINHOOD_MCP_URL:` and `ROBINHOOD_MCP_TOKEN:` **empty**; `refresh.py:151` then picks `source="yahoo"` → `_fetch_yahoo` (`refresh.py:65`), the keyless v8 chart API with a spoofed browser User-Agent. Local `.env` has no RH token either. The MCP fetch path's payload shape was "Confirmed live 2026-07-06" (`data.py:172`) — once, 16 days ago, and never exercised since. Unverified whether it still works.
- **Cache**: `data/*.csv` on `paper-state` is fresh through 2026-07-21 (verified via `git show`). The local checkout's copy ends 2026-07-15 — stale mirror; the branch is the source of truth.
- **No synthetic data on the live path.** The fallback chain (`data.py:62`) degrades MCP→Yahoo→*absent*, never to a made-up number. But failures are soft: a symbol Yahoo can't serve is skipped with a stderr line (`refresh.py:91`) and the run continues.
- **Silent-substitution risks that do exist**: `get_bars` (`data.py:47-49`) is cache-first and never refetches an existing CSV, so a stale cache is used without complaint; `update_cache` (`refresh.py:59`) keeps "last row wins" on date collisions, so one bad Yahoo bar silently overwrites a good one; `net_returns` (`backtest.py:44`) does `fillna(0.0)`.

## 6. Are the results real?

- **Forward strategy record: real, current-code, and tiny.** `journal/forward/mean_reversion/returns.csv` on paper-state = 8 rows, 2026-07-09→2026-07-20, total **+0.17%**, "Sharpe 2.57". Produced by the code as it stands (CI log output matches file content). Eight days is statistically nothing.
- **Forward agent record: real and vacuous.** 4 days, all 0.0. The decisions log (130 entries, 2026-07-20/21) is: 63× "parse-fail: held", 46× "parse-fail: AttributeError: 'NoneType'…" (the `re.search(...).group(0)` crash at `engine.py:157`, caught by the bare `except` at 164), 4× timeout — **113/130 (87%) are failures**, recorded indistinguishably from decisions. The remaining 17 genuine verdicts all said flat. The agent's "+0.00% return" is a record of API failures, not trading. Its self-written memory (`journal/agent_memory.md`) is content-free filler ("pre-schedule review times, e.g. every Friday at 2 PM").
- **Papertrade archive: reproducible but in-sample.** Latest run (2026-07-22): 284 days, 776 trades, +8.63%. It replays 2025-06→2026-07 — the same window on which mean-reversion+conviction was *selected* from 3 strategies × 4 overlays. `config.yaml` honestly admits the deflated-Sharpe CI lower bound "sits ~0". The Sharpe 1.82 headline is unverified in this audit and is selection-biased by construction.
- **Execution bias, systematic**: `CloseFill` (`papertrade.py:74`) and `backtest.net_returns` both fill at the *same close that generated the signal* — you cannot observe the close and trade at it. This flatters mean reversion specifically (buying the dip at the dip's own close). Costs are 1bp per side, no spread, no slippage model. Survivorship: the universe is today's mega-caps, chosen recently.
- **No benchmark anywhere.** Same window, SPY buy-and-hold: **+25.20%** vs the strategy's +8.63% (lower exposure, granted — but no output ever shows this comparison).
- Trustworthy files: the forward `returns.csv` files as records of what the code computed. Not trustworthy as evidence of edge: everything, until forward data is ~100× longer and fills are realistic.

## 7. Load-bearing and undefended

- **`except Exception: held` at `engine.py:164`** — total model failure is recorded as a decision. It has already silently consumed the entire agent track record.
- **Forward full-coverage freeze** (`forward.py:121`, admitted in its own comment): one symbol Yahoo stops serving and the record silently stops growing forever. No alert exists.
- **Yahoo** is a single unauthenticated dependency with no schema validation; per-symbol failures are stderr lines in a green CI run.
- **`paper-state` is the only copy of the track record** — gitignored on main, no backup, and appendable by anyone: `deploy_api/trigger-run.js` is deliberately unauthenticated ("blast radius is CI spam" — also blast radius is polluting the run archive).
- **No CI test gate** — the daily run executes whatever is on main, untested.
- Stale-cache silence in `get_bars` and last-row-wins merging in `update_cache` (see §5).

## 8. Over-built

- **`scripts/make_dashboard.py`: 1,354 lines — 40% the size of the entire `src/` tree** (3,332 lines), regenerated from scratch per run, to display 8 days of returns.
- **The agent stack** (`AgentEngine` + `memory.py` reflection + `learn.py` lessons): ~65 LLM calls plus a reflection call per day, with `max_retries=8`/45s timeouts (the 14–17-minute CI runs are mostly retry burn), to pick a value in {-1, 0, 1} from three scalars *the prompt itself computes* (`engine.py:127-147`). Ten lines of thresholds would outperform it — and would at least return parseable output.
- The guardrails/executor/broker/journal architecture: well-built, well-tested, and guarding a road with no traffic.
- `MarketSource`/`FillModel` protocols with one implementation each (`papertrade.py:29-34`, self-described "world-model-ready"); dual `TOOLS`/`OPENAI_TOOLS` schemas (`agent.py:46-96`) for the unscheduled loop.
- Credit where due: `BucketFilter`/`WinProbGate` were actually deleted after losing the bake-off (`overlay.py:89`) — real discipline, rare.

## 9. The five questions you should be asking

1. **Why is the LLM's live failure rate 87%, and why does a failure count as a decision?** Until parse-fails are fixed and *excluded*, every agent metric — including its clean 0.00% — is noise you're paying NVIDIA for.
2. **What sample size would change your mind?** At ~0.02%/day mean and daily ticks, distinguishing the conviction gate from zero takes on the order of *years* of forward data. What's the pre-committed kill/keep criterion, or is this record going to be read as confirmation regardless?
3. **If you ever flip `LIVE=true`, which program trades?** The safety story (guardrails, halt file, loss limits) lives in a codebase branch nothing schedules; the code that runs daily has no guardrails, no broker, no kill switch. The audit trail you'd show someone after a blow-up describes a system that wasn't running.
4. **Does the edge survive real execution?** Same-close fills at 1bp is the most flattering possible assumption for a dip-buying strategy. Re-run the bake-off with next-open fills and ~5-10bp all-in cost; if conviction-gated mean reversion still wins, you have something. Expectation: the margin shrinks badly.
5. **Why is "Robinhood" in the name?** No credential exists in CI or locally, the MCP payload shape was verified once on 2026-07-06, production data is Yahoo, and the one benchmark that matters (SPY +25.2% vs strategy +8.6% over the same window) appears nowhere in any output. Relatedly: the entire track record lives on one git branch with no backup and a public, unauthenticated append button.
